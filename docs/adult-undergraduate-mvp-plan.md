# 成人高考专升本个人备考系统——第一版规划方案

## 1. 文档定位

本文档用于对齐第一版产品范围和业务颗粒度，后续作为数据整理、页面设计和代码改造的依据。

第一版只面向个人使用，只覆盖江苏省和河南省的成人高考专升本，不建设通用招生平台，不加入智能择校推荐，优先跑通以下闭环：

```text
选择报考省份
  → 选择意向专业
  → 查看开设该专业的院校
  → 确认目标院校与专业
  → 自动匹配考试科目
  → 完成入学诊断
  → 计算目标分数
  → 生成或展示分阶段学习路线
  → 学习与阶段测试
  → 根据测试结果给出复习建议
```

## 2. 第一版目标

第一版需要解决四个核心问题：

1. 我可以在江苏、河南中的哪些省份报名。
2. 我想报某个专业时，有哪些院校可以选择，各院校的成本和录取难度如何。
3. 选定院校和专业后，需要考哪些科目，目前距离目标分数还有多大差距。
4. 如何以尽可能短的时间，从零基础稳定达到目标录取分数。

第一版以“辅助个人决策和备考”为目标，不替代省教育考试院及院校当年发布的正式招生政策。

## 3. 产品范围

### 3.1 第一版包含

- 江苏、河南两个报考省份，可多选。
- 根据当前日期自动确定参考年度。
- 成人高考专升本专业目录。
- 根据专业反查招生院校。
- 院校专业的历年录取分数、学制、学费、教学点和学位申请条件。
- 根据专业类别匹配考试科目。
- 入学诊断与分数差距分析。
- 科学计算目标分数。
- 计划模式和自主模式。
- 分阶段学习路线。
- 阶段测试和非强制复习建议。

### 3.2 第一版不包含

- 智能择校推荐、冲稳保排序和录取概率预测。
- 报名、缴费、准考证、成绩查询等流程提醒。
- 自动提交报名或对接考试院账号。
- 江苏、河南以外的省份。
- 自考专升本、普通专升本等其他考试类型。
- 社交、排行榜、课程交易等平台能力。

上述能力可以保留在后续版本规划中，但不进入第一版开发范围。

## 4. 核心业务流程

### 4.1 建立个人报考档案

#### 4.1.1 需要填写的信息

第一版只收集生成后续方案所必需的信息：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 可报考省份 | 多选 | 江苏、河南，至少选择一个 |
| 参考年度 | 自动生成，可手动修改 | 根据当前日期计算默认值 |
| 意向专业 | 单选或暂存多个候选 | 第一版主流程最终确认一个专业 |
| 学习模式 | 单选 | 计划模式、自主模式 |
| 每日可学习时间 | 计划模式必填 | 可按工作日、周末分别设置 |
| 预计开始日期 | 默认当天 | 用于计算学习周期 |

当前学历、毕业状态等信息第一版不用于复杂资格审核，可以作为备注信息保留。如果后续需要判断专业限制或报名资格，再升级为结构化字段。

#### 4.1.2 报考省份提示

省份支持多选，例如户籍在河南、工作在江苏时，可以同时保留河南和江苏两个选择。

界面提示文案建议为：

> 通常可在户籍所在地报名；在非户籍所在地报名时，可能需要有效居住证或一定期限的连续社保记录。具体材料和期限以该省当年成人高校招生公告为准。

系统可以在数据中保存“居住证、连续社保 3～6 个月”等参考信息，但不能把具体月数永久写死。不同省份、招生年度的要求可能变化，需要带上年度和官方来源。

#### 4.1.3 参考年度规则

默认规则：

```text
当前月份为 1～10 月  → 参考年度默认为当前年份
当前月份为 11～12 月 → 参考年度默认为下一年份
```

第一版将 10 月视为当年考试周期结束月份。年度切换点应保存为配置项，而不是散落在页面代码中；如果后续确认报名截止日或考试安排发生变化，只需要修改配置。

页面需要显示系统自动选择年度的原因，并允许手动切换年度。

#### 4.1.4 专业与考试科目的关系

档案阶段先选择意向专业，但此时不让用户手工填写考试科目。系统根据以下关系自动匹配：

```text
意向专业 → 专业类别 → 统考科目 → 可能存在的院校加试
```

更换专业后，考试科目、诊断结果、目标分和学习计划都需要重新计算，并在切换前提示用户受影响的数据。

### 4.2 根据专业查看招生院校

用户完成省份和专业选择后，系统按参考年度列出在所选省份招收该专业的院校。

每一所院校至少展示：

| 信息 | 说明 |
| --- | --- |
| 院校名称 | 成人高考招生院校名称 |
| 招生省份 | 江苏或河南 |
| 招生年度 | 明确到年份 |
| 专业名称 | 院校招生简章中的专业名称 |
| 专业类别 | 用于确定考试科目 |
| 历年录取分数 | 区分省控线、院校线、专业线 |
| 学制 | 例如 2.5 年，按招生简章保存 |
| 学费 | 标明单位、收费周期和数据年度 |
| 学习形式 | 按当年招生简章中的正式名称保存 |
| 教学点 | 名称、城市、地址或说明 |
| 学位申请条件 | 课程成绩、学位外语、论文等要求 |
| 数据来源 | 官方链接、发布日期、采集日期 |

#### 4.2.1 院校筛选

第一版只提供基础筛选，不做推荐算法：

- 按报考省份筛选。
- 按学费区间筛选。
- 按学制筛选。
- 按教学点城市筛选。
- 按是否有可确认的历史录取分数筛选。

用户可以进入院校详情比较信息，然后手动确认一个目标院校和专业。

#### 4.2.2 分数线类型

系统必须明确区分：

1. 省最低录取控制分数线。
2. 院校实际投档或最低录取分数线。
3. 院校具体专业最低录取分数线。

如果只取得省控线，不得将其显示为院校或专业录取线。缺失数据应直接显示“暂无可靠数据”，并标记数据可信度。

### 4.3 自动匹配考试科目

确认目标专业后，系统展示：

- 统考科目。
- 每科满分和总分。
- 科目对应的考试大纲年份。
- 推荐教材或资料版本。
- 是否可能存在院校加试。
- 数据来源和更新时间。

考试科目跟随专业类别变化。更换院校但专业类别不变时，统考科目通常不需要重新生成；如果目标院校存在加试，则追加院校加试信息。

### 4.4 入学诊断测试

诊断测试是生成目标分和最短学习方案的必要输入，包含以下五项：

1. 各科基础测试。
2. 知识点掌握度评估。
3. 做题速度评估。
4. 错题类型分析。
5. 可用学习时间评估。

基础测试不允许用户自行填写预估分数。系统按目标专业对应的考试科目组织诊断题库，每个诊断知识点提供 3～5 道单选题，由系统根据正确答案自动计算：

- 分科基础分数。
- 每个知识点的掌握度。
- 答题正确率和平均答题时间。
- 最薄弱知识点及优先复习建议。

用户只负责完成题目；可用学习时间直接读取报考档案中的计划设置。第一版题库可以采用抽样诊断，后续再按正式考试大纲扩充完整知识点树和题目数量。

#### 4.4.1 诊断输出

诊断完成后输出：

- 各科当前预估分数。
- 当前预估总分。
- 各知识模块掌握度。
- 主要失分类型。
- 各科目标分数差距。
- 预计所需有效学习小时数。
- 计划模式下的预计达标日期。
- 数据可信度；题量不足时提示结果仅供初步参考。

#### 4.4.2 零基础处理

用户可以选择“按零基础开始”，但仍建议完成一次简短摸底测试。即使基础较弱，摸底结果也能判断时间应该优先投入政治、英语还是专业基础课。

### 4.5 科学计算目标分

目标总分的基本规则：

```text
历史参考线 = 当前可获得的最细粒度录取线
              （专业线优先，其次院校线，最后省控线）

目标总分 = 历史参考线 + 30 分
```

当存在多个年份的数据时，不直接使用最低年份，建议使用：

```text
基础参考线 = 近三年有效录取线的较高值或加权值
目标总分   = 基础参考线 + 30 分
```

页面同时展示：

- 使用了哪一种分数线。
- 使用了哪些年份。
- 为什么增加 30 分。
- 当前预估分与目标分的差距。
- 由于招生人数和报考人数变化，目标分不构成录取承诺。

#### 4.5.1 分科目标分配

系统不能把总分差距平均分配到每一科，应结合诊断结果分配：

- 优先选择单位时间提分效率较高的科目和知识点。
- 保证各科基础题和高频题的得分。
- 对短期提升困难的模块设置合理上限。
- 分科目标之和应略高于目标总分，给临场波动保留空间。

### 4.6 学习模式

#### 4.6.1 计划模式

适用于希望系统自动安排每天学习任务的情况。

用户需要设置：

- 计划开始日期。
- 工作日每天可学习时长。
- 周末每天可学习时长。
- 每周休息日。
- 无法学习的特殊日期，可选。

系统自动生成：

- 预计达标日期。
- 每个阶段的开始和结束日期。
- 每日科目、知识点、学习时长和练习量。
- 每周复习和补漏时间。
- 阶段测试日期。
- 因任务延期而产生的后续调整建议。

如果按照用户提供的时间无法在考试前达到目标，系统应明确提示预计缺口，并建议增加每天学习时长、减少休息日或调整目标，而不是生成看似完整但无法完成的计划。

#### 4.6.2 自主模式

适用于用户自行决定每天何时学习，不需要系统生成日历任务的情况。

系统仍然提供：

- 完整的学习阶段。
- 各阶段需要完成的知识点和题量。
- 建议学习顺序。
- 阶段完成条件。
- 学习记录入口。
- 阶段测试。
- 测试后的复习建议。

自主模式不生成每天必须完成的任务，也不因某天未学习而重新排期。

### 4.7 分阶段学习路线

第一版统一采用四个阶段，具体内容按科目动态生成。

#### 阶段一：基础建立

目标：理解基础概念，能够完成高频基础题。

完成条件示例：

- 核心基础知识点完成率达到设定值。
- 基础题正确率达到阶段要求。
- 完成阶段一测试。

#### 阶段二：专项提分

目标：针对主要题型和薄弱模块集中提分。

完成条件示例：

- 高频题型完成专项练习。
- 薄弱知识点正确率明显提升。
- 完成阶段二测试。

#### 阶段三：真题训练

目标：适应真实题目结构、难度和答题节奏。

完成条件示例：

- 完成指定年份的真题。
- 能够在规定时间内完成试卷。
- 完成阶段三测试。

#### 阶段四：模考冲刺

目标：稳定达到目标分数，减少非知识性失分。

完成条件示例：

- 完成多次整套模拟考试。
- 总分和分科成绩达到稳定标准。
- 高频错题已至少复习一次。

### 4.8 阶段测试与动态纠偏

每个阶段结束后提供一次阶段测试。测试结果包括：

阶段测试与入学诊断保持同一原则：不允许用户自行填写测试分数。系统从当前阶段覆盖的科目和知识点中抽取选择题，用户完成作答后自动计算正确率、折算分和薄弱知识点。

- 总分和各科分数。
- 与阶段目标、最终目标的差距。
- 知识点正确率变化。
- 做题速度变化。
- 主要错题类型。
- 下一阶段建议。

当测试成绩过低时，系统建议复习相关知识点，但不强制阻止用户进入下一阶段。

用户可以选择：

- 接受建议，插入一组复习任务。
- 暂时忽略，继续下一阶段。
- 重新进行本阶段测试。

计划模式下，接受复习建议后重新计算后续日期；自主模式下，只更新推荐学习顺序。

## 5. 页面结构

第一版建议保留以下页面：

1. **我的报考档案**：省份、参考年度、意向专业、学习模式和时间设置。
2. **专业与院校**：先选专业，再展示江苏、河南对应招生院校。
3. **院校详情**：录取分数、学制、学费、教学点、学位申请条件和来源。
4. **我的报考目标**：确认目标院校、专业和考试科目。
5. **入学诊断**：完成五项诊断并生成基础画像。
6. **目标分分析**：展示录取参考线、目标总分和分科差距。
7. **学习路线**：展示四个学习阶段及当前所处阶段。
8. **今日学习**：仅计划模式显示自动生成的每日任务。
9. **阶段测试**：答题、交卷和结果分析。
10. **学习进度**：阶段完成度、预计分数和测试趋势。

自主模式下可以隐藏“今日学习”，直接从“学习路线”进入知识点和学习记录。

## 6. 核心数据模型

第一版至少需要以下数据对象：

```text
UserProfile                 个人报考档案
ProvinceEligibilityRule    分省、分年度报名资格提示
AdmissionYearConfig        参考年度和切换月份配置
Major                      专业基础信息
MajorCategory              专业类别
ExamSubjectRule            专业类别与考试科目的关系
Institution                院校基础信息
AdmissionProgram           某省某年度的院校招生专业
TeachingSite               教学点
TuitionRule                学费和收费方式
DegreeRequirement          学位申请条件
AdmissionScore             省控线、院校线或专业线
DiagnosticResult           入学诊断结果
ScoreTarget                目标总分和分科目标
StudyPlan                  学习方案
StudyStage                 学习阶段
StudyTask                  每日或阶段任务
StageTest                  阶段测试
TestResult                 测试结果和复习建议
DataSource                 官方来源和数据可信度
```

所有招生数据必须包含省份、招生年度和来源，避免把不同年度的数据错误地组合在一起。

## 7. 数据来源与可信度

数据来源优先级：

1. 江苏省教育考试院、河南省教育考试院。
2. 院校继续教育学院或成人教育学院官网。
3. 正式招生简章、招生专业目录和收费文件。
4. 第三方网站仅用于发现线索，不作为关键字段的唯一依据。

每条关键数据建议保存：

```json
{
  "province": "江苏",
  "year": 2026,
  "sourceType": "official",
  "sourceUrl": "https://...",
  "publishedAt": "2026-XX-XX",
  "collectedAt": "2026-XX-XX",
  "confidence": "verified"
}
```

对于暂无官方数据的下一招生年度，可以展示最近年度数据作为参考，但必须明确标记“当年招生计划尚未发布”。

## 8. 后续版本保留项

以下功能保留在文档中，但不进入第一版：

- 智能择校与冲稳保方案。
- 基于招生计划变化的录取概率预测。
- 报名、审核、缴费、准考证、考试和录取提醒。
- 征集志愿和调剂辅助。
- 更多省份。
- 课程资源推荐和题库扩充。
- 多用户、云同步和账号体系。

## 9. 第一版验收标准

第一版完成时，应能够完整演示以下场景：

1. 用户同时选择河南和江苏作为可报考省份。
2. 系统根据日期自动给出参考年度，并允许修改。
3. 用户先选择专业，系统分别列出两省开设该专业的院校。
4. 用户可以查看院校的历年分数、学制、学费、教学点和学位条件。
5. 用户确认一个目标院校和专业后，系统自动给出考试科目。
6. 用户完成按考试科目组织的知识点选择题，系统自动计算当前分数、知识点掌握度、目标分和分数差距。
7. 计划模式能够根据可用时间生成四阶段计划和每日任务。
8. 自主模式能够展示四阶段路线，但不自动安排每日任务。
9. 每个阶段结束后可以参加测试。
10. 成绩过低时系统给出复习建议，用户可以接受、忽略或重测。
11. 所有招生和分数数据都能追溯到省份、年度和来源。

## 10. 后续实施顺序

后续代码改造建议按以下顺序进行：

1. 整理并确认数据模型。
2. 获取江苏、河南当年招生专业、院校及考试科目数据。
3. 获取学制、学费、教学点、学位条件和历年分数数据。
4. 改造个人报考档案与“专业反查院校”流程。
5. 实现目标分计算和入学诊断。
6. 实现计划模式、自主模式和四阶段路线。
7. 实现阶段测试与非强制纠偏。
8. 最后进行页面整合和完整流程验收。
